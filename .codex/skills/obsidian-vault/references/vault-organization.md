# Vault 구조화 원칙

vault 구조를 손볼 필요가 있을 때 이 기준을 사용한다.
처음부터 거대한 체계를 들이기보다, 시간이 지나도 버틸 수 있는 가벼운 구조를 선호한다.

## 기본 입장

현재 사용자의 구조에서 출발한다.
사용자가 원하지 않으면 큰 프레임워크를 억지로 도입하지 않는다.
잘게 쪼갠 폴더 여러 개보다, 의미가 분명한 상위 영역 몇 개를 우선한다.

## 추천 상위 영역

vault가 아직 작거나 초기 단계라면 다음 정도가 무난한 출발점이다.

- `Philosophy/`: 가치관, 신념, 정체성, 세계관, 삶의 원칙
- `Study/`: 개발 공부, 면접 준비, 영어 공부, 독서 정리
- `Projects/`: 진행 중인 프로젝트와 종료된 프로젝트 기록
- `Ideas/`: 거친 발상, 실험, 메모, 미래 확장 씨앗
- `Reference/`: 참고 자료, 용어집, 큐레이션 자료
- `Maps/`: 허브 노트, 인덱스 노트

이 구조는 출발점일 뿐 규칙집이 아니다.
사용자의 표현 습관에 맞게 합치거나 이름을 바꿔도 된다.

## 권장 기본 폴더 구조

사용자가 문서 작성을 자주 요청하고, 분류 체계를 분명하게 유지하고 싶다면 다음 구조를 기본값으로 사용한다.

- `00 Maps/`
- `01 Philosophy/`
- `02 Study/Development/`
- `02 Study/Career/`
- `02 Study/English/`
- `03 Projects/`
- `03 Projects/Content/`
- `04 Ideas/`
- `05 Reference/`

이 구조의 의도는 다음과 같다.

- `00 Maps/`는 구조를 설명하고 연결하는 허브 영역
- `01 Philosophy/`는 개인 철학과 가치관을 담는 중심 영역
- `02 Study/`는 학습 자산을 주제별로 축적하는 영역
- `03 Projects/`는 실제로 진행 중인 실행 단위를 관리하는 영역
- `04 Ideas/`는 아직 실행 전인 생각과 발상을 보관하는 영역
- `05 Reference/`는 직접 사유한 글보다 보조 자료를 모으는 영역

## 문서 배치 절차

새 노트를 만들 때는 다음 순서를 따른다.

1. 같은 주제의 기존 노트가 있는지 찾는다
2. 그 노트가 속한 폴더가 이미 있으면 그 안에 작성한다
3. 반복 가능한 주제라면 새 하위 폴더를 만든다
4. 일회성 메모가 아니라면 적절한 상위 영역 아래로 분류한다
5. 루트에는 가급적 문서를 두지 않는다

예시:

- 유튜브 채널 기획 문서 -> `03 Projects/Content/YouTube/Channels/<채널명>/`
- 영어 표현 정리 -> `02 Study/English/`
- 삶의 태도에 대한 성찰 -> `01 Philosophy/`
- 막 떠오른 서비스 아이디어 -> `04 Ideas/`
- 링크 모음이나 참고 리스트 -> `05 Reference/`

플랫폼 아래 여러 단위가 생길 수 있는 경우는 한 단계 더 구조화한다.

예시:

- YouTube 플랫폼 루트 -> `03 Projects/Content/YouTube/`
- 채널별 문서 -> `03 Projects/Content/YouTube/Channels/<채널명>/`
- 여러 채널이 함께 쓰는 운영 메모나 공통 기준 -> `03 Projects/Content/YouTube/Shared/`

## 제목 규칙

노트 제목은 출처보다 의미를 드러내도록 짓는다.

좋은 예:

- `Dependency Injection`
- `왜 나는 속도보다 공예적 완성도를 더 중시하는가`
- `영어 면접에서 자주 쓰는 표현`
- `Project X Decision Log`

날짜는 시간이 주된 탐색 기준일 때만 앞에 붙인다.

- 데일리 노트
- 미팅 노트
- 진행 로그
- 저널

## 링크 규칙

링크는 미래 탐색을 돕는 경우에만 만든다.
기계적으로 모든 키워드에 링크를 걸지 않는다.

가치가 큰 연결:

- 개념과 선행 개념 연결
- 프로젝트 노트와 의사결정 노트 연결
- 성찰 노트와 관련 가치관 연결
- 학습 노트와 실전 예시 연결

특정 주제가 커지면 `Maps/` 아래에 허브 노트를 만들어 핵심 노트들을 묶는다.

## 리팩터링 규칙

구조 개선은 작은 단위로 나눠서 진행한다.
다음 중 하나를 우선 고려한다.

1. 과하게 비대한 노트를 둘이나 셋으로 분리하기
2. 흩어진 노트를 연결하는 허브 노트 만들기
3. 모호한 제목을 더 분명하게 바꾸기
4. 새 위치가 명확히 더 좋을 때만 노트 이동하기

사용자가 구조 개편을 명시적으로 원하지 않았다면 대규모 이동은 피한다.
